## Problem Statement

You're building Meridian for NVIDIA Spark Hack Toronto (deadline: Sunday May 31, 4pm). The idea doc is comprehensive, but has structural gaps that could cost you the win. The biggest: your V1 architecture reads as a deterministic data pipeline, not an agentic system. At a hackathon whose tagline is "Your Agents. Your Models. Your Edge," this is the #1 risk.

## What Makes This Cool

The combination of 6 city open data sources into a single property risk verdict is genuinely novel. No existing tool synthesizes heritage designation + flood zone + development pressure + building permits + RentSafeTO + transit proximity into one report. The "default prevention" framing turns a data aggregator into a mission-driven product.

- Llama 3.1 8B is the confirmed LLM
- 3-5 minute demo slot

## Premises (Validated)

1. Hidden cost framing is the right hook, but needs the agent reasoning layer to feel like AI rather than a calculator
2. 6 data sources is a strong differentiator for the "Toronto Open Data" criterion
3. Llama 3.1 8B needs careful prompting to avoid generic/templated output
4. The 4-agent architecture needs visible reasoning to qualify as "agentic"
5. GB10 usage needs a story beyond "inference box"
6. Pre-selected properties are fine for the scripted demo, but live capability is insurance

## Gaps Identified (Beyond the Idea Doc's Own Analysis)

### Gap 1: No Fallback Strategy for API Failures
The V1 pipeline hits 6+ external APIs in real-time. During a live demo, a single CKAN timeout or ArcGIS error stalls the entire system. The idea doc mentions no caching, retry logic, or graceful degradation.

**Fix:** Hybrid approach. Pre-cache results for your 3 demo properties. For live queries, implement 5-second timeout per source with graceful degradation ("Flood data unavailable for this session" rather than a crash). The agent should acknowledge missing data in its reasoning: "I was unable to verify flood zone status, so I'm flagging this as an unknown risk."

### Gap 2: Geocoding Dependency Unspecified
Agent 1 geocodes the address but the doc doesn't name the geocoder. Toronto addresses have quirks (unit numbers, "East"/"West" suffixes, numbered streets vs named streets). A wrong geocode silently breaks every spatial query downstream.

**Fix:** Use the City of Toronto's own Address Points dataset (available on Open Data Portal) as primary geocoder. Fallback to Nominatim/OSM. Log the coordinates in the agent's reasoning trace so judges can see: "Resolved '401 Richmond St W' to [43.6492, -79.3925]. Querying within 500m radius."

### Gap 3: LLM Hallucination on Sparse Data
When a property has few flags (no heritage, no flood, no permits), an 8B model may fabricate risks to fill space. This is catastrophic in a live demo if a judge knows the property.

**Fix:** Structured prompting. Pass the LLM a JSON object with explicit nulls for missing data. Instruct it: "If a field is null, say 'No concerns identified' for that category. Never speculate about data you were not given." Include a system prompt that says "You are a factual report generator. You may ONLY reference data provided in the structured input."

### Gap 4: No Build Schedule
12 features listed for V1 with no priority order or time allocation. If things go wrong at hour 8, what gets cut?

**Fix:** See Build Schedule below.

### Gap 5: "Agentic" Solution in Doc is Wrong
The doc proposes "conversational follow-up" as the agentic differentiator. That's a chatbot, not an agent. Judges know the difference.

**Fix:** The agent should demonstrate autonomous decision-making DURING the pipeline:
- **Planning step:** Agent 1 analyzes the address and property type, then DECIDES which data sources are relevant. "This is a condo, so I'll check RentSafeTO and building permits for assessment risk. This is ground-floor, so flood zone is a priority. Skipping heritage check because this is a 2019 build."
- **Reasoning trace:** Show the agent's chain-of-thought in the UI. Not just results, but WHY it queried what it queried.
- **Contradiction detection:** When data sources conflict (e.g., building is new but has many permits), the agent should flag this and explain why it matters.
- **Confidence scoring:** The agent rates its own confidence per flag (high/medium/low) based on data quality and recency.

### Gap 6: Pitch Script vs. Demo Strategy Mismatch
The "Pitch Script (Final)" opens with statistics. The "Demo Strategy" section says to open with Sarah's story. These contradict each other. The Sarah story is stronger.

**Fix:** Rewrite pitch to open with Sarah. Statistics become supporting evidence, not the lead. "Meet Sarah. She bought a condo for $750K. Here's what she didn't know. [demo]. Now here's why this matters at scale: 1.15 million Canadians..."

### Gap 7: NVIDIA Story is Generic
"Runs locally on GB10" is what every team will say. You need a specific claim.

**Fix:** Frame it as: "Financial AI MUST run on-device. Bank data can never leave the user's machine. The GB10's 128GB unified memory lets us run inference on sensitive financial data with zero cloud dependency. This is the use case edge AI was designed for." Then during demo, show the NIM container running locally and point to the privacy guarantee.

## Recommended Approach: Visible Agent Reasoning

> **Superseded (PRD_Backend_Build.md §0 D1):** The proposed topology below was the gap-analysis recommendation. The locked decision differs in one key place: **Agent 3 is deterministic (no LLM)** — all dollar math is pure Python. Agent 1 includes the LLM planning call; Agent 4 is the second and final LLM call. The "PROPOSED" diagram below is historical context; build to PRD_Backend_Build.md §1.

### Architecture Change

```
CURRENT (from idea doc):
  Agent 1: Intake + Geocode (deterministic)
  Agent 2: Property Risk Engine (parallel queries)
  Agent 3: Cost Computation (deterministic math)
  Agent 4: LLM Synthesis (Llama 3.1 8B)

PROPOSED (this doc — partially superseded, see note above):
  Agent 1: Planning Agent (LLM-powered)
    - Receives address + any known property metadata
    - DECIDES which data sources to query and WHY
    - Outputs a query plan with reasoning
    
  Agent 2: Data Retrieval (deterministic, parallel)
    - Executes the query plan from Agent 1
    - Returns structured results with data quality metadata
    - Flags timeouts/failures explicitly
    
  Agent 3: Analysis Agent (LLM-powered) ← SUPERSEDED: Agent 3 is deterministic
    - Receives structured data + Agent 1's plan
    - Detects contradictions, patterns, correlations
    - Computes costs (deterministic math wrapped in reasoning)
    - Rates confidence per finding
    
  Agent 4: Synthesis Agent (LLM-powered)
    - Generates the final report
    - References Agent 1's reasoning and Agent 3's analysis
    - Produces the verdict with explicit justification
```

### What This Changes for the Demo

Instead of showing: "Enter address → wait → get report"

You show: "Enter address → agent THINKS about what to check → agent EXPLAINS its plan → agent DISCOVERS findings → agent REASONS about what they mean → agent DELIVERS verdict with confidence scores"

The reasoning trace is visible in the UI. The judge sees the agent making decisions, not just fetching data.

### Key Implementation Detail: The Reasoning Trace UI

In Streamlit, add an expander or sidebar that shows the agent's reasoning in real-time:

```
[Planning] Analyzing property type... This appears to be a pre-1990 condo
  in a heritage neighbourhood. Prioritizing: heritage register, building
  permits (assessment risk), flood zone (proximity to Don River).
  Skipping: RentSafeTO (no score found = likely not a rental building).

[Retrieval] Querying Heritage Register... FOUND: Part IV designation (1987)
[Retrieval] Querying Building Permits... FOUND: 3 active permits (structural)
[Retrieval] Querying TRCA Flood Zone... CLEAR: Property is outside floodplain
[Retrieval] Querying Dev Applications... FOUND: 12 applications within 500m

  Heritage restrictions will increase renovation costs 40-60%.
  12 nearby dev applications suggest intensification pressure on property taxes.
  Confidence: HIGH (all data sources returned valid results).

[Verdict] RED FLAG: This property has material hidden costs...
```

THIS is what makes it agentic. The agent is visibly reasoning, not just formatting.

## Prompt Skeletons

### Agent 1: Planning Prompt
```
SYSTEM: You are a property risk planning agent. Given a Toronto address, infer the property type and decide which data sources to query. Output valid JSON matching the schema below. Be specific about WHY you chose each source. If a source is irrelevant, mark it SKIP with a reason.

Available sources: heritage_register, building_permits, trca_flood, dev_applications, rentsafeto

Rules:
- heritage_register: query for any building pre-2000 or in known heritage areas
- building_permits: always query for condos (assessment risk indicator)
- trca_flood: query for ground-floor or basement units, or properties near Don River/Humber River/Lake Ontario
- dev_applications: always query (standard 500m radius)
- rentsafeto: query only for residential rental buildings (apartment buildings, not condos)

USER: Address: {address}. Known metadata: {any listing details if available, otherwise "none"}.
```

### Agent 3: Analysis Prompt
```
SYSTEM: You are a property risk analysis agent. You receive structured data from multiple Toronto city data sources. Your job:
1. Identify findings with severity (red/yellow/green) and confidence (high/medium/low)
2. Flag contradictions between data sources (e.g., new building with many permits)
3. Compute the 10-year cost breakdown using the formulas provided
4. If a field is null or a source returned no data, say "No concerns identified" for that category. NEVER speculate about data you were not given.

Output valid JSON matching the schema provided.

USER: Query plan reasoning: {agent_1_reasoning}. Retrieved data: {agent_2_output_json}.
```

### Agent 4: Synthesis Prompt
```
SYSTEM: You are a property report synthesis agent. Generate a plain-English report for a home buyer. Reference specific data points from the analysis. Include:
1. One-paragraph summary (the verdict)
2. Hidden costs breakdown (dollar amounts)
3. Risk flags with explanations
4. 10-year cost comparison (3 mortgage scenarios)

Rules:
- Every claim must reference a specific data source finding
- Use dollar amounts, not percentages, for cost impacts
- If confidence is "medium" or "low", say so explicitly
- Do NOT fabricate data. Only reference findings in the input.
- Tone: direct, clear, protective of the buyer's interests

USER: Planning reasoning: {agent_1_trace}. Analysis findings: {agent_3_output_json}. Property: {address} at ${list_price}.
```

## Latency Budget

**Assumption:** Llama 3.1 8B on GX10 (Blackwell, 128GB unified memory) via NIM achieves ~80-120 tokens/second for inference.

| Step | Expected Latency | Max Acceptable |
|------|-----------------|----------------|
| Agent 1 (Planning, ~200 token output) | 2-3 sec | 5 sec |
| Agent 2 (6 parallel API calls) | 2-4 sec | 8 sec |
| Agent 3 (Analysis, ~400 token output) | 4-5 sec | 8 sec |
| Agent 4 (Synthesis, ~600 token output) | 6-8 sec | 12 sec |
| **Total pipeline** | **14-20 sec** | **33 sec max** |

**Validation checkpoint (Hour 1):** Benchmark a single NIM call. If >15 seconds for 300 tokens, merge Agent 3 + Agent 4 into one call and reduce max_tokens to 800.

**Demo pacing:** 20-30 seconds of "thinking" is fine if the reasoning trace is updating in real-time. The audience sees the agent working, not waiting. Use `st.status()` with expanding sections.

## Expanded Data Sources (Post-Research Update)

A deep research pass across all 1,052 datasets on Toronto Open Data identified 10 additional parameters that use the same CKAN API patterns already in Agent 2. These are documented in [`DATA_PARAMETERS.md`](./DATA_PARAMETERS.md) with confidence scores and build effort estimates.

Key additions for V1.5:
- **Residential Fire Inspections** (124,414 records, daily) — building-level fire code compliance, far more precise than intersection-level fire incidents
- **Heritage Formerly Listed** (~3,700 records) — Bill 23 delistings signal neighbourhood change
- **Building Violations** (48,035 records, daily) — open compliance issues transfer to buyer
- **Transit Dividend** (TTC GTFS) — $87K transport cost differential over 10 years

The spec (`meridian_spec_final.txt` v3) also introduced two composite inferred signals:
- **Maintenance Complexity Signal** — composite of building age + structural permits + RentSafeTO + heritage + flood (always Low confidence)
- **Future Tax Pressure Signal** — composite of dev app density + permit activity + heritage/zoning (always Low confidence, never MPAC prediction)
